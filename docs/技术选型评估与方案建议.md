# 崂应报价系统技术选型评估与方案建议

## 1. 结论概览

本文基于当前项目实际代码与 `docs/system-design.md` 进行评估，结论如下：

- 当前 **Python + Streamlit + pandas + openpyxl** 的主技术选型 **总体不需要推倒重来**，它与“本地电脑运行、1~2 位工厂技术人员使用、以 Excel 导入导出为核心”的场景是匹配的。
- 需要调整的重点 **不是改语言或直接改成前后端分离**，而是把现有实现从“能跑”收敛到“可维护、可扩展、可验证”。
- 建议优先考虑 **保留当前技术栈并做模块化重构**；如果后续对“安装体验、桌面感、权限控制、多用户协作”要求明显提高，再考虑升级到桌面化或服务化方案。

### 1.1 评估依据

本次判断主要基于以下现有文件，而不是脱离代码的抽象讨论：

- `docs/system-design.md`
- `01-脚本文件/sheet_metal_quote_full.py`
- `common_quote_export.py`
- `pages/2_崂应报价.py`
- `pages/3_新星报价.py`
- `报价系统首页.py`
- `app_home.py`
- `02-启动报价页面/启动多客户报价系统.bat`

## 2. 项目背景与评估前提

根据现有文档与代码，当前系统具有以下特征：

- 目标：自动化钣金报价，并支持不同厂商/客户的报价格式。
- 用户：工厂技术人员，人数少，通常为 1~2 人。
- 环境：主要在用户本地 Windows 电脑运行。
- 数据形态：以 Excel、CSV、截图识别结果为主，最终仍要导出 Excel 给客户。
- 业务特点：报价公式较多、字段映射复杂、不同厂商导出格式差异大，但核心计算能力高度相似。

这意味着系统的首要目标不是“互联网式高并发”，而是：

- 本地运行稳定；
- 报价计算准确；
- Excel 导入导出好用；
- 新增客户格式时改动小；
- 出问题时方便排查和回滚。

---

## 3. 当前技术选型是否合适

### 3.1 适合保留的部分

| 技术/做法 | 结论 | 原因 |
|---|---|---|
| Python | 保留 | 报价计算、表格处理、快速迭代都很适合；团队维护门槛低。 |
| pandas | 保留 | 批量表格计算、导入导出、字段清洗都合适。 |
| openpyxl | 保留 | 当前交付结果就是 Excel，且需要样式、公式、说明页、盖章等能力。 |
| 本地文件系统导出 | 保留 | 符合本地单机使用场景，不必为了“正规化”过早引入服务端存储。 |
| Streamlit | 暂时保留 | 对内部工具、表格驱动、快速搭界面非常高效，当前用户规模也能承受。 |

### 3.2 不建议现在就调整的方向

以下方向从“技术上能做”变成“现在应该做”，证据还不够：

- 不建议现在就改成 `React/Vue + FastAPI` 前后端分离。
- 不建议现在就上远程数据库或云端服务。
- 不建议为了“桌面应用外观”立刻整体迁移到重型桌面框架。

原因很直接：

- 当前用户少，本地单机为主，复杂部署反而增加维护成本。
- 报价系统的核心价值在“公式正确 + 模板适配 + 导出稳定”，不是多人在线协作。
- 如果过早服务化，会把精力从业务规则沉淀转移到部署、网络、权限、兼容性上。

---

## 4. 当前方案需要调整的地方

这里的“需要调整”，主要是指 **架构层面的收敛**，不是整套重做。

### 4.1 核心问题判断

| 问题 | 当前表现 | 风险 |
|---|---|---|
| 核心逻辑集中在大脚本 | `sheet_metal_quote_full.py` 过大，导入、计算、导出、UI 混在一起 | 难测、难改、容易牵一发而动全身 |
| 客户扩展方式偏复制式 | 新客户页面容易复制已有脚本再改 | 后续客户一多，公式和字段映射会分叉 |
| 客户页与核心逻辑耦合不清 | `runpy` 代理可兼容旧代码，但长期不是理想结构 | 调试和复用成本会越来越高 |
| 参数缺少本地持久化 | 刷新后单价参数可能恢复默认 | 用户体验差，也容易造成误报/漏报 |
| 缺少统一的“客户格式适配层” | 导入列名、导出格式、默认参数散落在不同文件 | 新增一个厂商时不确定要改哪些位置 |
| 安装与启动依赖本机 Python 环境 | `.bat` 内直接 `pip install` | 网络、权限、版本漂移会影响稳定性 |
| 自动化测试不足 | 关键公式主要靠人工验证 | 一旦改公式，回归成本高，容易漏边界 |

### 4.2 对当前技术栈的真实评价

当前系统的问题，**80% 不是因为选错了 Streamlit 或 Python**，而是因为：

- 业务逻辑还没有形成稳定的分层；
- 多客户扩展还没有抽象成统一模式；
- 本地交付和配置管理还没有产品化。

所以，“是否需要调整技术选型”的答案应该是：

**需要调整，但优先调整的是架构组织方式，而不是底层语言或整体产品形态。**

## 5. 可选方案

下面给出 3 种可选路线，按改动成本从低到高排序。

### 方案 A：保留当前技术栈，做模块化重构（推荐）

#### 核心思路

保留：

- Python
- Streamlit
- pandas
- openpyxl

重点新增或重构：

- `core/`
  - `config.py`：默认参数、路径、材质库、常量
  - `calculator.py`：纯计算函数
  - `importers.py`：Excel/CSV/OCR 导入与字段映射
  - `exporters.py`：Excel 导出、盖章、各客户导出协议
  - `validators.py`：字段校验、报错整理
- `customers/`
  - `laoying.py`
  - `xinxing.py`
  - `dingxin.py`
  - 每个客户只负责“字段映射 + 默认参数 + 导出模板 + 页面文案”
- `storage/`
  - 本地 `json` 或 `sqlite` 持久化单价参数、最近导出记录、用户偏好

#### 适用场景

- 未来 6~12 个月仍是 1~2 人本地使用为主；
- 近期最重要的是“新增客户更快、公式更稳、回归更容易”；
- 希望控制改动风险，不打断当前报价流程。

#### 优点

- 改动最小，能复用现有代码和知识。
- 能最快提升可维护性与扩展性。
- 对本地 Excel 场景最友好，学习成本最低。
- 可以逐步迁移，不需要一次性重写。

#### 缺点

- UI 体验仍受 Streamlit 边界限制。
- 真正的桌面应用体验仍然一般，仍会体现为本地 Web。
- 如果未来演变为多人协作系统，仍需继续升级。

#### 推荐度

**最高。** 这是当前场景下最平衡的方案。

---

### 方案 B：保留 Python 核心，UI 改为桌面应用

#### 核心思路

把“报价计算引擎”继续保留在 Python 中，但把界面从 Streamlit 替换为：

- `PySide6`：更像正式 Windows 桌面软件；
- 或 `Flet`：开发门槛相对较低，界面现代感比 Streamlit 强。

本地存储建议同步升级为：

- `SQLite`：保存单价参数、客户模板配置、导出历史、最近打开记录。

#### 适用场景

- 用户强烈希望双击就能用，不想看到浏览器和端口；
- 对“像正式软件”的界面体验有明确要求；
- 后续需要更强的本地文件管理、对话框、打印、批量任务反馈。

#### 优点

- 本地用户体验更自然，培训成本更低。
- 更容易做成单文件安装包或标准桌面软件。
- 对本地文件操作、进度条、复杂表单交互支持更强。

#### 缺点

- UI 重写成本明显高于方案 A。
- 维护复杂度上升，开发速度会比 Streamlit 慢。
- 团队需要熟悉桌面框架的事件模型和打包流程。

#### 推荐度

**中等。** 如果“交付体验”已经成为主要痛点，可以考虑；否则不建议先走这条路线。

---

### 方案 C：升级为本地服务化或轻量前后端分离

#### 核心思路

采用：

- 后端：`FastAPI`
- 前端：`React` / `Vue`
- 本地存储：`SQLite`
- 如果未来跨机器部署，再考虑 `PostgreSQL`

将系统拆成：

- 报价计算服务
- 客户模板配置服务
- 导入导出服务
- 前端操作界面

#### 适用场景

- 未来明确要变成多人同时使用；
- 需要账号、权限、审批、历史追踪、局域网共享；
- 计划把报价能力变成标准平台，而不只是内部工具。

#### 优点

- 架构边界清晰，更适合多人协作与长期演进。
- 前端体验和权限体系更容易做完整。
- 后续做审计日志、模板中心、统一配置更方便。

#### 缺点

- 对当前 1~2 人本地场景来说，明显偏重。
- 开发、部署、排障、维护成本都会显著上升。
- 一旦进入前后端分离，测试面、接口契约、环境管理都会复杂很多。

#### 推荐度

**当前不推荐。** 除非业务目标已经从“本地报价工具”升级为“多人共享报价平台”。

---

## 6. 三种方案对比

| 维度 | 方案 A：保留 Streamlit 重构 | 方案 B：Python + 桌面 UI | 方案 C：前后端分离 |
|---|---|---|---|
| 改动成本 | 低 | 中 | 高 |
| 上线风险 | 低 | 中 | 高 |
| 本地运行适配度 | 高 | 高 | 中 |
| Excel 场景适配度 | 高 | 高 | 中 |
| 新客户扩展效率 | 高 | 高 | 高 |
| 用户使用体验 | 中 | 高 | 中到高 |
| 多人协作能力 | 低 | 低到中 | 高 |
| 当前阶段推荐度 | 高 | 中 | 低 |

## 7. 推荐结论

### 7.1 结论一句话

**建议采用方案 A：保留当前技术栈，但把系统从“大脚本 + 多页面拼接”升级为“通用报价引擎 + 客户适配层 + 本地持久化”的结构。**

### 7.2 推荐原因

它最符合当前业务现实：

- 用户少；
- 本地运行；
- Excel 导入导出为核心；
- 报价准确性比界面华丽更重要；
- 新增客户格式会持续发生。

在这个前提下，方案 A 能以最小风险解决当前最关键的问题：

- 核心公式沉淀；
- 客户格式复用；
- 参数持久化；
- 回归验证能力；
- 后续是否升级到桌面/服务化的可演进空间。

## 8. 建议的落地路线

### 第 1 阶段：稳定化（优先）

目标：不改用户使用方式，先把系统变稳。

建议内容：

1. **先只做最小闭环**：优先抽离 `calculator.py`，让报价计算变成纯函数，并保证拆分前后结果一致。
2. 抽离 `importers.py` 与 `exporters.py`，把导入导出和 UI 分开。
3. 新建“客户配置对象”或“客户适配器”：
   - 默认列定义
   - 字段映射规则
   - 默认单价参数
   - 导出格式策略
4. 将单价参数、最近配置持久化到本地：
   - 简单起步可用 `JSON`
   - 只有在需要保存历史、多套模板、版本对比时，再升级到 `SQLite`
5. 为关键函数补最小必要的单元测试：
   - 材质识别
   - 重量计算
   - 材料费计算
   - 特殊加工费计算
   - 表头映射

建议的实际顺序如下：

1. 先拆计算核心；
2. 再收敛客户适配层；
3. 然后补最关键的回归测试；
4. 最后再做参数持久化与打包优化。

### 第 2 阶段：扩展化

目标：让新增客户格式变成可复制流程，而不是复制代码。

建议内容：

1. 统一客户适配接口，例如：
   - `get_input_schema()`
   - `normalize_import(df)`
   - `calculate(df, params)`
   - `export_quote(df, params)`
2. 将客户差异尽量收敛到配置或适配层。
3. 建立“新客户接入清单”：
   - 导入字段
   - 计算差异
   - 导出模板
   - 验证样例

### 第 3 阶段：产品化（按需）

只有当以下需求明确出现时，再考虑方案 B 或 C：

- 必须打包成真正桌面软件；
- 需要账号权限；
- 需要多人共享同一套报价规则；
- 需要统一模板中心与审批追踪。

## 9. 关键技术建议

### 9.1 关于本地持久化

建议不要继续完全依赖 `session_state` 保存关键参数。

建议：

- 轻量场景：用 `JSON` 保存当前单价参数、最近导出目录、用户偏好；
- 稍复杂场景：用 `SQLite` 保存参数版本、客户模板、导出历史、变更时间。

判断标准：

- 如果只是“记住上次参数”，`JSON` 就够；
- 如果要“多客户多模板多版本对比”，直接上 `SQLite` 更稳妥。

### 9.2 关于客户适配模式

建议把“不同厂商报价格式”看作 **适配器问题**，而不是“多复制几个页面”。

更合适的抽象方式是：

- 通用引擎负责计算；
- 客户适配器负责字段、默认值、导出格式；
- 页面只负责交互和展示。

这样后续新增客户时，优先新增一个适配器，而不是复制一整份脚本。

### 9.3 关于打包与交付

当前 `.bat + pip install + streamlit run` 的方式适合开发阶段，但不适合作为长期交付方案。

建议至少做两步收敛：

1. 固定依赖版本，避免每次启动临时安装最新包。
2. 后续视情况增加：
   - `venv` 初始化脚本；
   - 或 `PyInstaller` 打包；
   - 或把 Python 运行环境整体随项目分发。

这样可以明显减少“用户电脑环境不一致”导致的问题。

## 10. 性能与稳定性说明

在当前场景下，性能瓶颈大概率不在框架本身，而在以下位置：

- 大表导入时重复拷贝 `DataFrame`；
- 每次渲染都重复做迁移、格式整理、参数同步；
- Excel 导出时重复拼装样式和宽表；
- OCR 首次初始化过慢。

因此，不建议通过“更换前端技术栈”来解决性能问题，优先应做：

- 减少重复读取与重复计算；
- 将材质库、参数表、模板信息做本地缓存；
- 大批量导入导出增加进度提示；
- 将计算逻辑做纯函数化，便于压测与单测；
- 对 OCR 做按需启用，而不是默认进入主流程。

这类优化比直接改技术栈更有效，也更符合当前项目规模。

---

## 11. 最终建议

### 建议立即执行

- 保留 `Python + Streamlit + pandas + openpyxl`
- 以“模块化重构 + 客户适配器 + 本地持久化 + 自动化测试”为主线推进
- 将“新增客户”从复制页面，调整为新增适配器/模板

### 建议暂缓执行

- 重做为前后端分离系统
- 引入远程数据库
- 为了形式统一而整体迁移桌面框架

### 触发升级的条件

若后续出现以下任一情况，再考虑方案 B 或 C：

- 用户数量明显增加；
- 需要跨电脑共享数据；
- 需要账号/权限/审批；
- 需要完整历史追溯与参数版本管理；
- 需要标准桌面安装包与更强本地交互体验。

## 12. 待执行验证清单

本文档已根据当前代码和设计文档整理，但尚未经过业务负责人逐条确认。建议按以下步骤验证：

1. 对照 `docs/system-design.md` 与当前代码，确认“当前技术栈”和“实际目录结构”描述无误。
2. 由实际使用该系统的技术人员确认：当前主要痛点究竟是“公式维护/扩客户”，还是“桌面体验/安装体验”。
3. 如果决定采用方案 A，先选一个客户页面做小范围试拆：
   - 将计算逻辑抽到独立模块；
   - 保持导出结果不变；
   - 用同一份样例数据比对拆分前后报价结果。
4. 手工验证现有主流程：
   - 执行 `python -m streamlit run app_home.py`
   - 进入崂应报价页与新星报价页
   - 导入样例数据并导出 Excel
   - 确认导出字段、总价、样式与当前业务预期一致
5. 若后续加入本地持久化，补充边界验证：
   - 修改单价后刷新页面，参数应保留；
   - 切换客户模板后，不应污染其他客户配置；
   - 导出历史记录应可追溯，不应覆盖旧数据。
